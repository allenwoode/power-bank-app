# AGENTS.md

## 目标

本文件面向进入本仓库协作的代理与内部开发者，帮助快速理解项目结构、工作边界和改动风险。目标是让协作在不破坏现有移动端能力的前提下，稳定推进需求、修复问题和补全文档。

## 项目概览

- 项目是一个基于 `React Native 0.81`、`Expo 54`、`Expo Router`、`NativeWind` 的智能充电宝 App。
- 核心业务包括 BLE 蓝牙连接、设备管理、WiFi 配网、地图定位、OTA 升级、通知配置、账户体系和中英文国际化。
- 代码和文档以中文语境为主；新增注释、说明、协作文档优先使用中文，代码标识符沿用现有英文命名习惯。

## 先看哪里

开始任何改动前，优先阅读这些文件，先理解再动手：

- `README.md`：项目背景、目录、命令、构建说明。
- `package.json`：脚本入口、依赖、工具链。
- `app/_layout.tsx`：全局 Provider、主题、启动页和 onboarding 路由分流。
- `lib/ble-service.ts`：BLE 单例服务，属于高风险核心模块。
- `i18n/index.ts` 与 `i18n/lang/*.json`：语言初始化和文案资源。

如果任务只涉及某个功能区，再继续下钻对应页面和组件，不要一开始做大范围重构。

## 目录职责

- `app/`：基于 Expo Router 的页面目录。
- `app/(welcome)/`：首次启动引导和 onboarding 流程。
- `app/(auth)/`：登录、密码登录、注册。
- `app/(tabs)/`：主 Tab 页面，当前重点是首页设备列表与“我的”。
- `app/(device)/`：设备相关页面，如添加、详情、地图、OTA。
- `app/(settings)/`：应用设置、通知、个人资料、安全相关页面。
- `components/`：复用 UI 和业务组件；`components/home/`、`components/ui/` 已按场景分层。
- `hooks/`：通用 Hook，如主题、导航防抖。
- `lib/`：跨页面复用的核心服务；当前最重要的是 BLE 服务。
- `context/`：全局上下文，如 onboarding、通知。
- `i18n/`：国际化初始化和语言资源。
- `utils/`：轻量工具函数。
- `patches/`：第三方依赖补丁，默认不要改。
- `android/`、`ios/`：原生工程目录；只有在确有原生需求时才进入。

## 代码约定

- 使用 TypeScript，项目开启了 `strict`，不要通过滥用 `any` 回避类型问题。
- 路径别名统一使用 `@/`，不要引入风格混杂的相对路径层级。
- 遵循现有格式化结果：
  - 单引号
  - 分号
  - `printWidth: 80`
  - `useTabs: true`
- 样式优先沿用 `NativeWind` 的 `className` 方案；只有动态值或组件库限制时再配合 `style`。
- 不要随意改动现有中英文混合的注释风格；新增注释只写必要信息，重点解释“为什么”，不要解释显而易见的代码。

## 页面与交互约定

- 路由按分组组织，新增页面时优先放入已有分组，不要破坏 `(welcome)`、`(auth)`、`(tabs)`、`(device)`、`(settings)` 的边界。
- `app/_layout.tsx` 中已经承载了启动页、主题、Provider 和 onboarding 路由切换逻辑；修改这里时要评估全局影响。
- 现有代码已引入 `useDebouncedNavigation` 处理重复点击导航，新增高频跳转入口时优先复用，而不是直接裸调 `router.push()`。
- 视觉与交互应延续现有移动端风格，不要引入只适合 Web 的布局假设。

## 数据与国际化约定

- 用户可见文案必须进入国际化资源，不要把中文或英文文案硬编码在页面里。
- 新增文案时同步更新 `i18n/lang/zh.json` 和 `i18n/lang/en.json`，并保证 key 命名清晰、可复用。
- `AsyncStorage` 已用于 onboarding、语言和设备列表等状态持久化；改动相关 key 时要检查读写链路和兼容性。
- 涉及设备存储结构变更时，优先考虑兼容已有本地数据，避免无提示破坏用户缓存。

## BLE 与设备相关高风险区

以下改动默认属于高风险操作，除非任务明确要求，否则避免顺手重构：

- `lib/ble-service.ts` 的连接、扫描、写入、清理逻辑。
- 蓝牙权限、后台能力和原生插件配置。
- `patches/react-native-ble-plx+3.5.0.patch`。
- `app.json`、`eas.json`、`android/`、`ios/` 中与构建、权限、原生插件相关的配置。

处理这些区域时请遵守：

- 小步修改，尽量一次只改变一个行为。
- 保留现有错误分类和兜底逻辑，除非你已经完整理解调用链。
- 不要在未验证的情况下调整权限声明、构建 profile、bundle id、package name 或地图/BLE 插件配置。
- 如果任务不涉及原生能力，不要为了“顺手整理”去改原生目录。

## 开发命令

常用命令如下：

```bash
npm install
npm start
npm run android
npm run ios
npm run web
npm run lint
npm run format
npm run pre-release
```

补充说明：

- `npm run pre-release` 会生成许可证信息并递增版本号，只有在确实准备发版时再用。
- Android 本地构建脚本会进入原生工程，属于高成本操作，除非任务需要，不要随意触发。
- 当前仓库自动化测试覆盖较少，变更后至少要做和改动直接相关的检查。

## 修改原则

- 先局部理解，再局部修改；不要在单个任务里顺手做大范围清理。
- 优先复用已有组件、Hook、样式模式和页面布局。
- 新增能力时，先看相邻目录是否已有类似实现，再决定是否抽象。
- 用户未要求时，不要修改版本号、构建配置、证书相关字段、包名、EAS profile。
- 用户未要求时，不要覆盖工作区中与你任务无关的本地改动。

## 提交与验证建议

- 提交信息遵循 Conventional Commits，仓库已配置 `@commitlint/config-conventional`。
- 完成改动后，至少运行与改动相关的验证；常见最低标准是 `npm run lint`。
- 如果没有运行验证命令，或因为环境限制无法运行，要在交付说明里明确写出。
- 如果改动涉及：
  - 路由：说明入口、返回路径和分组归属。
  - 国际化：说明新增的 key 和语言资源是否已同步。
  - BLE / 权限 / 原生配置：说明风险点和实际验证范围。

## 代理协作提醒

- 这个仓库更适合“小而准”的改动，不适合未经确认的大重构。
- 发现工作区已有未提交改动时，默认视为用户正在进行中的工作，不要回滚或覆盖。
- 对可疑行为保持克制：删文件、重置配置、重写路由结构、改原生工程，都需要充分理由。
- 如果你发现任务表述与当前代码现状明显冲突，先基于仓库事实给出判断，再继续实施。

## 一句话原则

先读清上下文，沿用现有模式，小步修改高风险区，并把验证范围说清楚。
